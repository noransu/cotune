# CoTune：一键启动前后端联调环境，告别端口战争

## 一、你是不是也遇到过这些问题？

做前后端联调的时候，每次开发前的"启动仪式"大概是这样的：

1. 打开一个终端，`cd` 到前端目录，`npm run dev`
2. 再开一个终端，`cd` 到后端目录，`mvn spring-boot:run`
3. 然后想起来前端 proxy 配置里写的后端端口不对，改一下，重启前端
4. 后端加了 `context-path`，前端请求全部 404，排查半小时发现是路径不匹配
5. 下班前忘了杀进程，第二天来启动直接 `EADDRINUSE: address already in use`
6. 前端说"我调了接口没数据"，后端说"我没收到请求"，互相看日志对了半天

如果你对这些场景感同身受，CoTune 就是为了解决这些问题而做的。

## 二、CoTune 是什么

CoTune 是一个桌面开发工具，把**前端项目、后端项目、反向代理、浏览器、终端**整合在一个窗口里。

核心能力：

- **一键启动**：前端、后端、代理服务全部一键启停
- **智能路由**：自动扫描 Spring Boot 源码中的接口注解，生成代理规则，不需要手动配 proxy
- **请求可视化**：实时看到每个请求走到了哪里——命中了哪个后端接口、返回了什么状态码
- **零侵入**：不需要修改前端或后端任何一行代码

创建一次项目配置，之后每次开发只需点一下 **Start**。

## 三、快速上手

### 3.1 安装

从内网下载对应平台的安装包：

- Apple Silicon (M 系列芯片)：`cotune-0.1.0-arm64.dmg`
- Intel Mac：`cotune-0.1.0-x64.dmg`

> **macOS 安全提示**：由于应用未签名，首次打开时 macOS 会阻止运行。请右键点击应用图标 → 选择「打开」→ 在弹窗中再次点击「打开」即可。

### 3.2 创建项目

打开 CoTune 后，左侧是项目面板。点击顶部的 **"+"** 按钮创建一个新项目。

**第一步：配置前端**

点击 **Select Frontend** 按钮，选择前端项目目录。CoTune 会自动识别：

- 项目名称（从 `package.json` 读取）
- 启动命令（如 `umi dev`、`npm run dev`、`vite`）
- 开发服务器端口（如 `8000`）

如果自动识别不准确，可以手动修改。

**第二步：配置后端**

点击 **Add Backend** 按钮，选择后端项目目录。CoTune 会自动检测 Spring Boot 项目，包括：

- 多模块项目会自动列出所有子模块，勾选你需要的
- 自动识别端口（从 `application.yml` / `application.properties` 读取）
- 自动识别启动命令

需要关注的配置项：

| 配置项 | 说明 | 示例 |
|--------|------|------|
| **Port** | 后端服务端口 | `8080` |
| **Context Path** | `server.servlet.context-path` 的值，没有就留空 | `/trumpet-user` |
| **API Prefix** | 前端请求统一加的网关前缀，CoTune 转发时会自动剥离 | `/api` |
| **External** | 勾选后 CoTune 不启停该服务，只负责代理（适用于 IntelliJ 手动启动的后端） | - |

**第三步：设置代理端口**

默认是 `9000`。启动后，你在浏览器访问 `localhost:9000` 即可，CoTune 会自动判断每个请求应该转发到前端还是后端。

**第四步：保存**

点击 **Save**，项目配置完成。配置会自动持久化，下次打开 CoTune 不需要重新配。

### 3.3 一键启动

在左侧项目列表中选中项目，点击 **Start** 按钮。CoTune 会依次执行：

1. **扫描后端路由** — 解析 Java 源码中的 `@GetMapping`、`@PostMapping`、`@RequestMapping` 等注解，生成代理规则
2. **启动后端进程** — 执行配置的启动命令（标记为 External 的跳过）
3. **启动前端进程** — 执行前端 dev server
4. **启动反向代理** — 在代理端口上监听，根据路由规则分发请求
5. **自动打开浏览器** — 在内嵌浏览器中打开 `localhost:9000`

整个过程通常在 10-30 秒内完成，取决于项目启动速度。

> **端口冲突？不用管。** 如果目标端口被旧进程占用（比如上次没杀干净），CoTune 会在启动前自动清理，不会再出现 `EADDRINUSE` 的问题。

左侧项目卡片上会实时显示每个服务的状态（Starting → Running），你也可以对单个服务独立启停。

### 3.4 使用内嵌浏览器

启动完成后，CoTune 会自动切换到浏览器页签。顶部有一个地址栏，功能和 Chrome 一样：

- 输入 URL 回车导航
- 前进 / 后退 / 刷新按钮
- **F12** 打开 DevTools（独立窗口）

你日常开发就在这个浏览器里操作就行，所有请求会经过 CoTune 的代理服务器，API 请求自动路由到后端。

### 3.5 查看路由映射和请求日志

地址栏右侧有一个 **"N Routes"** 按钮（N 是当前识别到的路由数量），点击展开侧边面板。

**Routes 标签页**

按 Controller 类名分组展示所有被识别的后端路由：

- 每条路由显示 HTTP 方法（GET / POST / PUT / DELETE）和路径
- 支持搜索过滤——输入关键词快速定位接口
- 点击 Controller 名称可以折叠/展开

这个视图帮你快速回答：**"后端到底有哪些接口？这个路径有没有被识别到？"**

**Logs 标签页**

实时展示经过代理的每一个请求：

- **蓝色** = 命中了后端路由，已转发到后端
- **灰色** = 没有匹配到后端路由，走的前端
- 显示状态码和耗时

这个视图帮你快速回答：**"我的请求到底走到了哪里？为什么没到后端？"**

### 3.6 使用内嵌终端

主界面底部是一个完整的终端，自动打开在当前项目目录下。你可以在这里执行 `git`、`npm`、`mvn` 等日常命令，不需要额外开终端窗口。

终端会话在你切换浏览器页签和回到主页之间保持不变——不会因为切了一下 tab 就丢失终端内容。

### 3.7 停止项目

点击 **Stop** 按钮，所有服务（前端、后端、代理）一键停止。也可以在项目卡片上对单个服务单独启停。

## 四、常见使用场景

### 场景 1：前端 + IntelliJ 启动的后端

后端同学习惯在 IntelliJ 里运行 Spring Boot 服务，不想改工作流。

**做法**：在 CoTune 中添加后端时，勾选 **External** 选项，填好端口号（比如 `8080`）。点击 Start 后，CoTune 会扫描后端源码的路由注解生成代理规则，但不会尝试启停后端进程。前端请求 `/api/trumpet/xxx` 自动代理到 8080，其他请求走前端 dev server。

两边各用各的工具，互不干扰。

### 场景 2：多模块后端项目

后端是一个多模块的 Maven 项目（比如有 `trumpet_server_applet`、`trumpet_server_admin` 等子模块）。

**做法**：选择后端项目根目录时，CoTune 会自动扫描并列出所有包含 `@SpringBootApplication` 的子模块。你可以选择需要的模块添加，每个模块独立配置端口和启停。路由扫描会精确到每个子模块的 Controller 层。

> **注意**：选择目录时要选到正确的层级。如果路由扫描结果为 0，通常是目录选错了——检查一下是不是选成了上级目录。

### 场景 3：联调排查——"请求到底走到了哪里？"

前端说"我调了这个接口没数据"，后端说"我没收到请求"。

**做法**：打开 CoTune 的 **Logs** 面板，找到对应的请求：

- 如果显示为蓝色（命中后端路由），检查后端逻辑
- 如果显示为灰色（未匹配），说明这个路径没有被识别为后端路由，请求走到了前端 dev server——这时候检查路由面板，看看路径有没有被扫描到
- 如果状态码是 404 或 502，结合转发目标端口判断是后端没启动还是接口路径不对

几秒钟定位问题，不用两边对日志。

### 场景 4：后端加了 context-path

后端在 `application.yml` 里配了 `server.servlet.context-path: /trumpet-user`，但前端请求路径不带这个前缀。

**做法**：在 CoTune 的后端配置中填写 **Context Path** 为 `/trumpet-user`。CoTune 扫描路由时会自动加上这个前缀来匹配请求，转发时保留完整路径。你不需要改前端的任何 proxy 配置。

如果前端请求统一加了一个网关前缀（比如 `/api`），在 **API Prefix** 字段填写 `/api`。CoTune 会在匹配路由时考虑这个前缀，转发到后端时自动剥离。

## 五、注意事项

1. **External 后端需要先手动启动**：CoTune 只代理，不管 External 服务的启停。确保 IntelliJ 里的服务已经跑起来再用 CoTune 访问。

2. **后端目录要选对**：多模块项目选择子模块目录（包含 `src/main/java` 的那个），不要选成了项目根目录的上级。路由扫描是在指定目录下递归找 `*.java` 文件。

3. **代理端口不要和前后端端口冲突**：默认 `9000`，确保这个端口没有被其他服务占用。

4. **目前深度支持 Spring Boot**：路由自动扫描目前支持 Spring Boot 的 `@RequestMapping`、`@GetMapping`、`@PostMapping`、`@PutMapping`、`@DeleteMapping`、`@PatchMapping` 注解。其他后端框架（Express、Go 等）暂时需要手动配置。

5. **配置会自动保存**：项目配置存储在本地，下次打开 CoTune 自动加载，不需要重复配置。
